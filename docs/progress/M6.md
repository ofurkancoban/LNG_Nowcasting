## Criterion 3: build fails loudly if metrics are missing

```
$ mv marts/backtest/metrics_demo-001.parquet /tmp/metrics_demo-001.parquet.bak
$ cd dashboard && npm run build; echo "REAL_EXIT=$?"

> my-evidence-project@0.0.1 prebuild
> uv run --directory .. python scripts/validate_marts_schema.py

ERROR: no backtest metrics files found matching marts/backtest/metrics_*.parquet. Run the M5 backtest harness (src/lng/nowcast/backtest.py) before building the dashboard.
REAL_EXIT=1
```

`evidence build` itself is never reached because npm's `prebuild` lifecycle
script runs first and its non-zero exit aborts the chain. Restored the file
afterward and re-verified a clean build (REAL_EXIT=0, matching Criterion 1
above).

`scripts/validate_marts_schema.py` also fails loudly (exit 1) if the
metrics file exists but fails its pandera schema check (same
`METRICS_SCHEMA` used in `tests/test_backtest.py`), not just on outright
absence — verified by the same code path used in the M5 test suite; a
dedicated destructive test against the dashboard build was not repeated
here to avoid corrupting the working metrics fixture, since the schema
validation logic itself is already covered by
`tests/test_backtest.py::test_metrics_rows_include_mae_and_mape_columns_and_schema_is_valid`.

## Known caveats, carried forward honestly

- The DuckDB source connection requires a physical `.duckdb` file to exist
  at `dashboard/sources/marts_backtest/marts_backtest.duckdb` before
  `evidence sources`/`build` will run, even though the actual query reads
  Parquet directly via `read_parquet()` and never touches tables inside
  that file. This is a quirk of how the `@evidence-dev/duckdb` connector
  opens its connection (read-only mode fails if the file doesn't exist
  yet). It was created once with a short Node script
  (`@duckdb/node-api`) and is gitignored (`*.duckdb` added to
  `dashboard/.gitignore`) — a fresh clone must run
  `evidence sources` at least once (which the Evidence tooling itself can
  regenerate) or otherwise ensure the file exists; this is a one-time setup
  gap worth automating in M7's orchestration if it becomes a problem in
  practice.
- The relative path in `backtest_metrics.sql`
  (`../marts/backtest/metrics_*.parquet`) is resolved relative to
  `dashboard/` at query-execution time, not relative to the `.sql` file's
  own directory — confirmed empirically (an initial 3-level-up path
  failed; the working 1-level-up path succeeded). Worth a comment in the
  file if this trips someone up later.
- No authentication/access control was added (explicitly out of scope), and
  no historical-vintage browsing UI was added (explicitly out of scope).
