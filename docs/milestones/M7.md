# M7: Orchestration, data quality checks, and alerting

## Objective

Run the pipeline continuously and reliably via GitHub Actions, with
automated data quality checks and loud failure alerts.

## Deliverables

- `.github/workflows/ingest-alsi.yml` (daily cron, timed after the 19:30 CET
  / 23:00 CET ALSI publication windows per docs/data_sources.md)
- `.github/workflows/nowcast-build.yml` (staging/marts rebuild + backtest,
  triggered after ALSI ingestion completes)
- `src/lng/quality/checks.py` (data quality assertions: raw AIS freshness
  per vessel, ALSI schema validation, marts row-count sanity checks)
- `tests/test_quality_checks.py`
- Long-lived AIS WebSocket ingester runs as a supervised process (systemd
  unit, restart-on-failure) on a dedicated VPS, not on GitHub Actions cron
  (which cannot hold a persistent connection). GitHub Actions remains
  responsible for the daily ALSI ingestion and marts/dashboard rebuild
  workflows only, and for a periodic health-check job that verifies the VPS
  ingester's last-write timestamp is recent (see quality checks below).
- `deploy/systemd/lng-ingest-aisstream.service` (unit file: restart policy,
  environment file reference for the AISStream API key, working directory)
- `deploy/README.md` (VPS provisioning steps: Python/uv install, systemd
  unit install, log rotation, how to check service status)

## Acceptance criteria

- `actionlint .github/workflows/*.yml` exits 0.
- `pytest tests/test_quality_checks.py` passes offline against fixture data,
  including at least one case that deliberately fails a check (e.g. a stale
  vessel or malformed ALSI row) and asserts the check raises/reports rather
  than passing silently.
- The workflow YAML for `nowcast-build.yml` has an explicit `needs:` or
  equivalent ordering dependency on the ALSI ingestion job, verifiable by
  grepping the yaml.
- At least one workflow step exists whose failure triggers a non-zero exit
  code that would fail the GitHub Actions run (verifiable by inspecting the
  yaml for `continue-on-error: false` or its absence, which is the default).
- `systemd-analyze verify deploy/systemd/lng-ingest-aisstream.service` exits
  0 (valid unit file syntax and `Restart=` policy set).
- A GitHub Actions health-check workflow polls the VPS ingester's
  last-write timestamp (e.g. via a lightweight status endpoint or the age
  of the newest raw partition file) and fails the run if it exceeds a
  defined staleness threshold.

## Out of scope

- Any specific third-party alerting integration (Slack/email/PagerDuty)
  beyond a failed CI run being visible in the GitHub Actions UI, unless the
  user specifies one.

## Dependencies

M1 through M6 (this orchestrates the full pipeline built by prior
milestones).
